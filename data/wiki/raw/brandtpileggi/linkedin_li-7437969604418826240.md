Most people use Claude Code like a chatbot.

Describe what you want, get some code, fix the bits that are wrong, repeat. That works for small stuff. It falls apart on anything complex.

I use five slash commands that turn Claude Code into a structured engineering workflow.

/brainstorm is where I develop an idea. It explores the codebase, asks me questions, and proposes approaches with trade-offs. If the feature needs multiple tasks it writes a sprint with dependencies, so each task can be specced individually.

/spec-task produces functional requirements through iterative conversation. What to build, not how. The output is a self-contained document a completely fresh session can pick up and understand.

/plan-task consults expert subagents that research online documentation, API specs, local repos for dependent libraries, and examples I've built before. Then it writes a phased implementation plan with testing strategy and anti-patterns to avoid.

/implement-task executes phase by phase. At every phase boundary it updates the task docs, runs tests, and commits. When the context compacts (and it will), the next session reads those docs and picks up exactly where it left off.

/review-task audits the code against the spec. Mechanical check first, then expert subagent validation of every issue found.

None of this is radical. It's just software engineering. Specs before code. Plans before implementation. Reviews before merge. The only difference is Claude Code does most of the legwork.
