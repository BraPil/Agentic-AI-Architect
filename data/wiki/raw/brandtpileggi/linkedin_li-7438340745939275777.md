Garry Tan, CEO of Y Combinator, just shared his Claude Code setup.

4,000+ lines of code. Full tests. Under 1 hour.

Here's what makes his approach DIFFERENT:

He doesn't ask Claude to write code first.

He asks it to review the plan. Like a staff engineer would before any changes are made.

One prompt forces Claude through a 4-stage structured review:

1️⃣ Architecture
System design, component boundaries, dependency graphs, scaling risks, security holes.

2️⃣ Code Quality
DRY violations flagged aggressively. Error handling gaps. Technical debt hotspots. Over-engineered vs under-engineered areas called out.

3️⃣ Tests
Coverage gaps across unit, integration, e2e. Missing edge cases. Untested failure modes. "Too many tests > too few."

4️⃣ Performance
N+1 queries. Memory usage. Caching opportunities. High-complexity paths.

For EVERY issue found, Claude must:

⚡ Describe the problem with file and line references
⚡ Present 2-3 options including "do nothing"
⚡ Specify effort, risk, impact, and maintenance burden
⚡ Make an opinionated recommendation
⚡ Ask for approval before proceeding

Two modes built in:

BIG CHANGE: full review, max 4 issues per section
SMALL CHANGE: one focused question per section

His 5 engineering preferences baked into the prompt:

1️⃣ DRY is non-negotiable
2️⃣ Well-tested code is non-negotiable
3️⃣ "Engineered enough" — not hacky, not over-abstracted
4️⃣ Handle MORE edge cases, not fewer
5️⃣ Explicit over clever. Thoughtfulness over speed.

The real insight:

Review before implementation. Tradeoffs before code. Feedback before action.

For small teams without a senior engineer on every PR: design the review process into your AI.

Are you still letting Claude Code skip the review? 👇
