Karpathy's LLM workflow went viral. I've been running an advanced version since mid-2025.

1,400 records. 440 signals. Producing content for NVIDIA, Google, CodeRabbit, and MongoDB.

Here's what 9 months in production taught me that a demo can't:

Obsidian is great for notes. It's terrible for memory.

1️⃣ You can't query markdown
"Show me every post that scored above 50 and ran on a Wednesday."
In a database: one query. Milliseconds.
In Obsidian: read every file. Hope Claude catches it. Pray the formatting is consistent across 9 months of entries.
At 50 notes this works. At 1,400 records it's burning tokens on irrelevant context every session.

2️⃣ You can't traverse relationships
"Which topics perform best with which audience segments?"
A database traverses this in one query.
Obsidian's wikilinks look like relationships. They're not. You can't query the structure. A folder of linked files is not a graph database any more than sticky notes on a wall are a relational schema.

3️⃣ No schema enforcement
Monday: Claude formats a source as ## Source Name - URL
Tuesday: **Source Name** (URL)
Wednesday: Source, see link
Good luck parsing that programmatically. Good luck even finding all three entries when you search.

4️⃣ Scale kills you silently
My signal library: 440 data points across 18 categories. Dumping that into context every session is expensive and crowds out actual work. Every token spent on "memory" is a token not available for the task.

5️⃣ Multiple agents break everything
I run scoring, sequencing, and scheduling agents on the same data. Two agents writing the same markdown file simultaneously = silent data corruption. Databases handle concurrent access natively. Markdown doesn't.

Memory system for cross-session context.

The .md files in my system contain instructions.
The databases contain the knowledge.

CLAUDE.md is a config file. Not a brain.

It started in Obsidian. It didn't stay there.
Your system shouldn't either.

20 minutes of database setup saves 6 months of workarounds.
